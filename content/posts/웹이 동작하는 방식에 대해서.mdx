# PM이 왜 웹이 동작하는 방식에 대해서 알아야 하는가?

구름에서 재직할때 강연을 듣다가 우연히 웹이 동작하는 방식에 궁금증이 생겨서 그때 찾아 봤었던 지식들을 한번 글로 옮겨 보려고 한다.

그 전까지는 웹이 동작하는 방식을 단순히 Domain 주소 -> DNS -> IP 호스트 주소로 변환 -> 서버에서 호출하는 방식으로만 알고 있었는데, 조금 더 깊이 있게 알아보고 싶어서 조금 찾아본 내용을 여기에 옮겨 담아 보려고 한다.

## 웹 사이트 접속

기본적으로 우리 유저들은 **"웹 브라우저"**라고 하는 애플리케이션을 통해서 특정 웹사이트에 접속한다.

웹사이트는 단순히 로컬에 저장이 되어 있어 **"짜잔"**하고 나타나는게 아니라 당연하게도 특정 서버에 저장되어 있는 정보를 기반으로 우리가 볼수 있는 형태의 웹사이트로 가공해 준다.

### DNS 호출
1. 유저는 브라우저에서 우리가 흔히 알고 있는 www. 로 시작하는 도메인 주소를 호출한다.
2. 그러면 클라이언트에서는 DNS (Domain Name System)라고 하는 시스템에 접속하려고 하는 도메인의 IP 호스트 주소를 요청한다.
3. 만약 요청했던 DNS 서버에서 IP 주소의 기록을 가지고 있다면 다음과 같이 작동한다.

| 순서  | 질의자                             | 응답자          | 종류  | 쿼리의 내용             |
| --- | ------------------------------- | ------------ | --- | ------------------ |
| 1   | Client                          | ns1.daum.net | 질의  | www.daum.net       |
| 2   | ns1.daum.net에서 자신의 레코드를 확인 (성공) |              |     |                    |
| 3   | 클라이언트                           | ns1.daum.net | 응답  | www.daum.net의 IP주소 |
DNS에는 DDNS나 DNS64, DNS over TLS 등등 다양한 방법들이 있지만 이 글은 DNS에 대한게 아니라 웹이 동작하는 방식에 대해서만 다루기 위한거니 다른 글에서 별도로 다루겠다. 

4. 클라이언트에서 다음과 같이 IP 주소를 받았다면 IP 주소에 있는 서버를 찾아 간다.

### 3 Way Handshake
4. 이때 TCP 프로토콜로 통신하기 위해 데이터 전송 전 상호 연결을 수립할 수 있는 3 Way Handshake 절차를 가지게 된다.
5. 클라이언트는 특정 번호를 가지고 서버에 연결을 요청하는 SYN 패킷을 전송한다.
6. 서버에서 잘 받은걸 확인하면 서버에서 다시 클라이언트로 패킷을 전달한다. 이게 바로 HTTP Request이다.
7. 웹브라우저는 이 데이터를 받게 되는데 이걸 HTTP Response라고 한다.

### Parsing
4. 웹브라우저는 받은 데이터를 유저에게 보여줘야 하는데, W3C명세에 따라 HTML과 CSS를 해석하게 된다. 이 작업을 Parsing이라고 한다.

### DOM Tree
4. 브라우저의 렌더링 엔진은 HTML을 Parsing하여 DOM Tree라고 하는 것을 생성한다. 
5. 만약 렌더링 엔진이 HTML을 Pasing하다 스타일 태그를 만나게 되면 작업을 중단하고 CSS Parsing을 위해서 CSSOM Tree라고 하는 것을 생성한다. 
6. CSS Parsing이 끝나면 HTML Parsing이 중단되었던 시점부터 작업을 재개한다.
7. 그러다가 데이터에 script 태그 (우리가 흔하게 알고 있는 Javascript)를 만나게 된다면 Parsing을 다시 중단하고 JS Engine에게 제어 권한을 넘긴다.
8. Script의 경우에는 추상 구문 Tree인 Abstract Syntax Tree를 만들게 된다.
9. 브라우저는 최종적으로 DOM Tree와 CSSOM Tree를 합쳐서 Redner Tree를 생성하게 된다.
10. Render Tree를 만드는 과정을 Construction이라고 명한다. 
11. 다음 과정은 Layout 이라고 하는 과정을 거치는데 이 과정은 Render Tree의 노드들을 화면의 올바른 위치에 표시하는 작업을 말한다.
12. 다음은 Composition 작업을 수행한다. Composition은 노드들의 레이어를 순서대로 구성하는 작업이다. Z-index가 낮은 요소를 먼저놓고 그 다음에 높은 요소를 놓는 작업이다.
13. Layout -> Paint -> Composition의 과정을 Operation이라고 한다.
